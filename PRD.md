## 一、背景与问题陈述

### 1.1 行业背景

根据[国家税务总局通报案例](https://www.nbd.com.cn/articles/2025-09-19/4067650.html),近年MCN机构因未依法代扣代缴旗下达人/主播个人所得税被处罚的案例频发。典型案例显示某MCN机构3年间向170名主播发放报酬1.6亿元,代缴个税却为0。其根本原因在于:

1. **达人身份信息核验缺失**:机构在向达人结算时,未对达人提供的姓名、身份证号、银行卡号进行真实性核验,导致申报数据错误或无法申报。
2. **所得性质认定混乱**:同一达人可能涉及"工资薪金""劳务报酬""经营所得"三类不同税目,人工判定易出错。
3. **税额计算复杂**:劳务报酬涉及加成征收(2万-5万加征五成、5万以上加征十成),手工计算极易出错。
4. **申报数据格式繁琐**:自然人电子税务局要求标准化的批量申报模板,从业务系统到申报系统的数据转换成本高。

### 1.2 政策依据

- 《中华人民共和国个人所得税法》及实施条例
- 《个人所得税扣缴申报管理办法(试行)》(国家税务总局公告2018年第61号)
- 《网络直播营销管理办法(试行)》
- 互联网平台从业人员个税新政(2025年10月1日实施)——累计预扣法适用劳务报酬

### 1.3 目标用户痛点

| 角色 | 主要痛点 |
| --- | --- |
| MCN财务 | 每月手工核对几十/上百名达人身份信息,核算劳务报酬税额耗时长、易出错 |
| 税务专员 | 自然人电子税务局批量导入模板格式严格,业务系统数据需手工拼接 |
| 合规负责人 | 难以追踪每位达人的代扣代缴是否到位,稽查风险高 |

## 二、产品定位与目标

### 2.1 产品定位

一款面向 MCN机构 / 平台方的 **达人身份核验 + 个税预扣预缴自动化 SaaS 工具**,核心解决"身份核验—税额计算—申报数据生成"全链路自动化问题。

### 2.2 核心目标

| 目标 | 量化指标 |
| --- | --- |
| 减少财务人工录入 | 单次结算流程从 30 分钟降至 3 分钟 |
| 降低申报差错率 | 税额计算 100% 符合税法规则,差错率 0 |
| 留痕可审计 | 每笔核验、每次计算、每份申报均留存日志 |
| 易扩展 | 接入真实"三要素/四要素"核验API后无需改前端 |

### 2.3 非目标 (Out of Scope)

- 不替代正式的自然人电子税务局申报系统(本工具仅生成符合格式的申报数据,实际报送仍走税局通道)
- 不处理增值税、企业所得税
- 不对接达人个人的银行账户进行实际打款

## 四、功能模块需求 (核心)

### 模块 M1 —— 达人档案管理

**职责**:维护达人基础信息,作为后续所有模块的数据源。 推荐以小红书达人库为主要数据源,人工录入仅作为补充手段。

#### 字段定义

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 姓名 | string | 是 | 真实姓名,与身份证一致 |
| 身份证号 | string | 是 | 18位,前端做基础校验;主键 |
| 手机号 | string | 否 | 11位,四要素核验时使用 |
| 银行卡号 | string | 否 | 四要素核验/打款使用 |
| 所得类型 | enum | 是 | `salary`/`labor`/`business`/`platform` (工资薪金/劳务报酬/经营所得/互联网平台) |
| 签约关系 | enum | 是 | `employee`/`contractor`/`studio` (员工/经纪合约/工作室) |
| 备注 | text | 否 | 自由文本;自动导入时会写入来源标识(如“来源: 小红书 xhs-1001 · @美妆张三”) |
| 核验状态 | enum | 系统 | `unverified`/`verified`/`failed` |
| 创建时间 | timestamp | 系统 | |

#### 功能要求

- **F1.1 从小红书一键批量导入(主要入口,v1.1)** 
  从小红书 MCN 后台拉取全部达人,以身份证号为主键做 upsert。 用户可在弹窗内为本次导入设置默认所得类型/签约关系;拉取后返回拉取/新建/更新/跳过四个计数。

  **冲突保护策略**: 已存在的达人只补全原本为空的字段,不覆盖人工修订过的姓名/手机/银行卡,避免自动同步覆盖人工校正。

- **F1.2 手动单条新增** 
  仅适用于小红书未覆盖的边缘场景(如跨平台达人、临时合作者)。 UI 上以次要按钮出现,出现在弹窗中。

- **F1.3 列表查询 + 来源可追溯** 
  分页 + 关键词搜索(姓名/身份证);列表展示“来源备注”列,每条记录的出处(小红书/双源比对/手动)清晰可查。

- **F1.4 删除/编辑** 
  单行删除;后续可增加软删除 + 历史记录。

#### 与双源比对的关系

- **本模块的导入**: 只依赖小红书达人库一个源,迅速建档。
- **双源比对(参见第十章)**: 需要与个税平台交叉校验时调用。 两者互补:达人档案 = 低门槛快速入档;双源比对 = 高信任交叉校验。

### 模块 M2 —— 身份核验引擎

**职责**:对达人提交的身份信息真实性进行核验。

**接口设计**(后端纯净接口,不依赖任何第三方实现):

```
POST /api/verify
  Request:
    {
      mode: "two" | "three" | "four",  // 二要素/三要素/四要素
      name: string,
      idCard: string,
      mobile?: string,
      bankCard?: string
    }
  Response:
    {
      success: boolean,
      passed: boolean,
      reason: string,
      providerTraceId: string  // 第三方核验流水号(预留)
    }
```

**功能要求**:
- F2.1 默认提供 **Mock 模拟器**:基于身份证校验码算法(GB 11643-1999)做基础校验,模拟通过/不通过结果
- F2.2 通过 `KycProvider` 接口抽象,支持后续接入阿里云/腾讯云/翼支付等真实三/四要素 API,**前端无需改动**
- F2.3 每次核验记录留痕,可在前端查看核验历史

**身份证算法校验规则**(Mock使用):
- 长度18位,前17位为数字,末位为数字或X
- 第7-14位为合法日期(YYYYMMDD)
- 末位校验码符合 GB 11643-1999 加权和算法

### 模块 M4 —— 个税计算引擎(核心)

**职责**:根据达人所得类型与金额,计算预扣预缴税额。

#### M4.1 劳务报酬 —— 按次预扣预缴(传统模式)

依据[劳务报酬计算公式](https://www.gerensuodeshui.cn/index_lwbc.html):

```
若 收入 ≤ 4000:
    应纳税所得额 = 收入 - 800
若 收入 > 4000:
    应纳税所得额 = 收入 × (1 - 20%)
```

**预扣率表(三级超额累进)**:

| 级数 | 预扣预缴应纳税所得额 | 预扣率 | 速算扣除数 |
| --- | --- | --- | --- |
| 1 | ≤ 20000 | 20% | 0 |
| 2 | 20000 < x ≤ 50000 | 30% | 2000 |
| 3 | > 50000 | 40% | 7000 |

```
应纳税额 = 应纳税所得额 × 预扣率 - 速算扣除数
```

#### M4.2 工资薪金 —— 累计预扣法

```
本期应预扣预缴税额 = (累计应纳税所得额 × 预扣率 - 速算扣除数) - 累计已预扣预缴税额

累计应纳税所得额 = 累计收入 - 累计减除费用 - 累计专项扣除 - 累计专项附加扣除 - 其他扣除
减除费用 = 5000 × 在职月份数
```

适用 **综合所得 7 级超额累进税率表**(年度):3% / 10% / 20% / 25% / 30% / 35% / 45%。

#### M4.3 经营所得 —— 5级超额累进(年度)

| 级数 | 全年应纳税所得额 | 税率 | 速算扣除数 |
| --- | --- | --- | --- |
| 1 | ≤ 30000 | 5% | 0 |
| 2 | 30000 < x ≤ 90000 | 10% | 1500 |
| 3 | 90000 < x ≤ 300000 | 20% | 10500 |
| 4 | 300000 < x ≤ 500000 | 30% | 40500 |
| 5 | > 500000 | 35% | 65500 |

#### M4.4 互联网平台累计预扣法 (2025-10-01 起)

```
本期应预扣预缴税额 = (累计收入 - 累计费用 - 累计减除费用) × 预扣率 - 速算扣除数 - 已预扣
累计费用 = 累计收入 × 20%
累计减除费用 = 5000 × 连续取得报酬月份数
```

**功能要求**:
- F4.1 引擎为纯函数模块,输入 → 输出,**不依赖数据库、不依赖HTTP**
- F4.2 单元测试覆盖每条税率边界值
- F4.3 计算结果含完整解释字段(便于前端展示推导过程)

### 模块 M6 —— 批量任务

**职责**:打通 M1-M5,实现"上传 Excel → 一键完成核验+计算+导出"。

**流程**:

1. 用户上传 Excel(姓名/身份证/手机/银行卡/所得类型/本期收入/所属期)
2. 系统逐行调用 M2 核验 → M4 计算

4. 用户可下载申报CSV / 失败明细CSV

**功能要求**:
- F6.1 单批次最多 1000 条
- F6.2 支持失败行重试
- F6.3 任务异步执行,前端轮询任务状态

## 五、架构设计 (关键)

### 5.1 三层解耦架构

```
┌────────────────────────────────────────────────┐
│  L1: 前端 UI 层 (React + shadcn/ui)            │
│  仅负责展示与交互,通过 fetch 调用 REST API    │
└────────────────────────────────────────────────┘
                       │ HTTP/JSON
┌────────────────────────────────────────────────┐
│  L2: API 网关层 (Express Routes)               │
│  仅负责参数校验、路由分发、统一响应包装        │
│  不含业务逻辑                                  │
└────────────────────────────────────────────────┘
                       │ 函数调用
┌────────────────────────────────────────────────┐
│  L3: 核心功能模块层 (纯 TypeScript 模块)       │
│  ├─ taxEngine     (个税计算,纯函数)            │
│  ├─ kycProvider   (身份核验,接口抽象)          │
│  ├─ talentRepo    (达人档案,数据访问)          │
│  ├─ incomeRepo    (收入登记,数据访问)          │
│  ├─ filingBuilder (申报数据生成)               │
│  └─ batchRunner   (批量调度)                   │
└────────────────────────────────────────────────┘
                       │
┌────────────────────────────────────────────────┐
│  L4: 持久层 (SQLite, 通过 Drizzle ORM)         │
└────────────────────────────────────────────────┘
```

### 5.2 解耦原则的落实

| 原则 | 实现方式 |
| --- | --- |
| 模块间低耦合 | 每个核心模块单独文件,通过 TS 接口契约通信。`taxEngine` 不知道数据库存在;`kycProvider` 不知道前端存在 |
| UI 与逻辑分离 | UI 只通过 `/api/*` 调用后端,**所有业务规则在 L3**。更换 UI(替成 Vue / 移动端)无需改动 L3 |
| 第三方可替换 | `KycProvider` 是接口,默认实现 `MockKycProvider`,未来接 `AliyunKycProvider` 只新增文件 |
| 计算可测试 | `taxEngine` 是纯函数,无副作用,直接单元测试 |

## 十、增量需求 v1.1 —— 双源比对模块

### 10.1 业务场景

小红书等内容平台的真实业务中,达人身份信息存在 **两套数据源**:

| 数据源 | 含义 | 权威性 |
| --- | --- | --- |
| **A. 个税平台导出 Excel** | 财务/税务从自然人电子税务局或扣缴客户端导出 | 税务权威源 |
| **B. 小红书达人库** | 达人在平台注册时绑定的实名信息 | 业务权威源 |

两套数据可能因人工录入、改名、换号等原因产生差异,直接结算会导致申报失败或申报错误人。

### 10.2 流程设计

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 上传税务 Excel │ -> │ 拉取小红书库 │ -> │ 自动双源比对 │ -> │ 人工裁决冲突 │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
                                                                    │
                                                                    ▼
                                              ┌──────────────────────────────┐
                                              │ 一键并入主档 → 进入代扣流程  │
                                              └──────────────────────────────┘
```

### 10.3 比对结果四象限

比对引擎按身份证号(主键)+ 姓名 进行匹配,产出 4 类:

| 类型 | 含义 | 默认动作 |
| --- | --- | --- |
| **MATCH** | 两源都有,且关键字段一致 | 直接并入主档 |
| **CONFLICT** | 两源都有,但姓名/手机/银行卡不一致 | 进入人工裁决 |
| **TAX_ONLY** | 只在税务 Excel 中存在 | 标记为"未在平台签约",可选择忽略或新建 |
| **PLATFORM_ONLY** | 只在小红书库中存在 | 标记为"未申报",可选择补录 |

### 10.4 关键字段比对规则

| 字段 | 比对方式 |
| --- | --- |

| 姓名 | 严格相等(去空格后) |
| 手机号 | 严格相等 |

### 10.5 数据源适配层(解耦关键)

export interface TalentSource {

比对引擎 `reconcileEngine.calculate(sourceA, sourceB)` 不关心数据来源,只接收两个数组。**未来增加抖音/B站达人源,只需新增一个 Source 实现,引擎与 UI 零改动。**

## 十一、增量需求 v1.1.1 —— 达人档案自动化导入

### 11.1 背景与决策

v1.0 初版只提供逐条表单录入达人档案,在中等及以上规模的 MCN 机构(达人 100+ 名)场景下人工输入压力大、质量低。 达人信息本身来自小红书 MCN 后台,应该以“从源头同步”为主要路径,手动补充为辅。

### 11.2 业务场景

- **MCN 运营人员** 接手新达人批次,期望一键同步 → 随后只处理例外。
- **达人信息变更** (手机/银行卡更新)后,可重复点击导入,系统以身份证号为主键增量合并,不重复创建。
- **人工修正后不被覆盖**: 如某位达人名字手动从“李思”改为“李思诗”,下次同步不会被平台源覆写。

### 11.3 交互设计

「达人档案」顶部操作栏调整为三区:
1. **主操作**(蔓延型主色): 「从小红书批量导入」—— 弹窗形式,可设默认所得类型/签约关系。
2. **次要操作**(outline): 「手动新增」—— 弹窗形式,表单字段同原。
3. **搜索框**: 姓名/身份证关键词搜索。

列表上方增加一条“自动化建档”提示条,引导用户使用推荐路径,并提示可进入「双源比对」做个税交叉校验。

### 11.4 后端接口

| 方法 | 路径 | 请求体 | 返回 |
| --- | --- | --- | --- |
| POST | /api/xhs/import | `{ defaultIncomeType?, defaultRelation? }` | `{ total, created, updated, skipped, failures[] }` |

**处理逻辑**:
1. 调用 `xhsTalentSource.fetch()` 拉取平台源全量达人。
2. 对每一条达人以 `idCard` 为主键查询 talents 表:
   - 不存在 → 创建,使用请求体中的默认所得类型/关系,备注写入“来源: 小红书 {sourceId} · {nickname}”。
   - 存在 → 只补全 mobile/bankCard 中原本为空的字段,姓名优先保留已有值。
3. 返回计数以供前端马上反馈。

### 11.5 与后续模块的关系

- 导入后达人 `kycStatus` 默认 `unverified`,需在「身份核验」模块进一步调用四要素核验。
- 若需与个税平台交叉校验,成果会进入「双源比对」中 PLATFORM_ONLY 象限(因为税务 Excel 中缺失)。

### 11.6 接口隔离原则在本增量中的体现

- **仅新增 1 个后端路由**: `POST /api/xhs/import`。
- **复用现有适配器**: 未新增任何 talentSource 实现,复用 `xhsTalentSource`。未来接入小红书开放平台/企业微信/抖音企业号等平台源时,仅需增加 N 个同接口的 source 实现,接口 + 路由不变。
- **前端 API 集中于 `lib/api.ts`**: 增加 `TalentApi.importFromXhs`,组件零直接 fetch。

## 十二、增量需求 v1.2 —— 三合一重构（详情抽屉 + 双源验证 + 批量同步）

### 12.1 背景

v1.1.1 上线后用户反馈：

1. 顶部 `MCN 机构 · 达人个税申报 Agent · v1.1` 过于冗余，MCN 机构名与版本号不应占据主位。
2. 达人表格只能看到脱敏后的身份证与手机号，看不到实名认证时上传的身份证照片、小红书账号详情、合作类型、变更历史。
3. 「身份核验」与「双源比对」二者本质上都是以实名信息为中心的验证动作，重复设为两个 Tab。
4. 「收入登记」需要逐笔手工输入，不够自动化；希望从小红书同步达人时一并拉取当期结算流水。
5. v1.1 「批量处理」仅是 CSV 粘贴，与「小红书同步」边界不清。

### 12.2 需求定义

#### 12.2.1 顶部品牌区简化

- 去掉“MCN 机构 ·”前缀与“v1.1”版本号，只保留主标题「达人个税申报 Agent」+ 一句定位说明。

#### 12.2.2 达人详情抽屉（新增）

表格任一行点击 → 右侧抽屉展示：

| 区块 | 内容 |
| ---- | ---- |
| 实名认证 | 身份证正/反面照片（Mock 使用占位图，实现对接 OSS 安全链接）、姓名/身份证/手机/银行卡脱敏信息 |
| 小红书账号档案 | 账号昵称、粉丝量级、粉丝数、合作类型、注册时间、实名认证时间、平台源 ID、签约关系 |
| 小红书结算流水 | 当期 × 达人维度的明细表（业务类型/结算单号/结算时间/税前金额/合计） |
| 近期核验记录 | 最近 5 条核验日志摘要（状态 + 模式 + 原因 + 时间戳） |
| 来源备注 | 档案的来源说明（小红书同步 / 双源比对 / 手动新增） |

#### 12.2.3 合并为「双源验证」

- 删除独立的「身份核验」与「双源比对」两个顶级 Tab。
- 新增「双源验证」顶级 Tab，内部使用子 Tab 切换两种颗粒度：
  - **单条核验**：针对个别达人的二/三/四要素实时核验（沿用原 KycPanel）。
  - **批量比对（税务 × 小红书）**：上传税务 Excel，与平台交叉勾稽（沿用原 ReconcilePanel）。

#### 12.2.4 「批量同步」升级（原「批量处理」）

重命名为「批量同步」，提供两种路径：

- **从小红书一键同步（默认 / 推荐）**: 选择所属期 + 默认参数 → 拉取本期结算流水 → 自动核验 → 自动计税 → 自动入库。面板充分展示：本期预览统计、6 个同步结果卡、逐人明细表。
- **自定义粘贴（兑底）**: 保留 v1.1 的 CSV 粘贴入口，用于无法从小红书获取流水的特殊场景。

#### 12.2.5 收入登记增加同步入口

- 「收入登记」顶部新增「从小红书同步本期收入」按钮，点击弹窗输入所属期后一键同步，成功后记录出现在下方明细表中。
- 原手工表单调整为「手工补录单笔收入」，限定为零散/调整场景。

#### 12.2.6 顶级 Tab 顺序调整

```
达人档案 → 双源验证 → 收入登记 → 税额试算 → 批量同步 → 申报导出 → 核验日志
```

该顺序与财务人员月度实际作业动作顺序一致：先梳达人、再交叉验证、手工零散补录后试算、月末一键同步、最后导出、随时查看日志。

### 12.3 后端接口增量

| 新增路由 | 说明 |
| ---- | ---- |
| `GET /api/xhs/incomes?period=YYYY-MM&idCards=...` | 预览小红书结算流水（不入库） |
| `POST /api/batch/xhs-sync` | 一站式同步：拉达人 + 拉流水 + 汇总 + 核验 + 计税 + 入库，返回 summary + results |

上述均复用现有核心模块（`xhsTalentSource.fetchIncomes()` + `runBatch()`），未新增业务计算逻辑。

### 12.4 数据模型微调

- `xhsTalentSource.SourceTalent.extra` 扩展为：
  ```ts
  {
    xhsNickname?: string;
    followers?: number;
    registeredAt?: string;
    verifiedAt?: string;
    idCardFrontUrl?: string;  // Mock://... 占位；生产环境为 OSS 预签名 URL
    idCardBackUrl?: string;
    contractType?: string;
    tier?: string;            // 头部/腰部/尾部达人
  }
  ```
- `XhsIncomeRecord` 新接口：`{ recordId, idCard, period, grossAmount, bizType, settledAt }`。
- 主库 `talents` 表结构不变。`extra` 仅存在于平台源返回体中，详情抽屉按身份证号按需拉取，避免入库冷数据。

### 12.5 架构与解耦原则落实

- **业务逻辑零重复**：`/api/batch/xhs-sync` 内部复用 `runBatch()`，未新增任何核验/计税代码。
- **UI 重组不伤及逻辑**：`VerifyPanel` 仅以子 Tab 组合 `KycPanel` 与 `ReconcilePanel`，两者代码零修改；未来双源验证业务可独立演进。
- **API 封装层不跨越**：任何面板均仅通过 `lib/api.ts` 调用后端，`SyncPanel` / `IncomePanel` / `TalentDetailDrawer` 零裸 fetch。
- **详情抽屉按需加载**：抽屉 `enabled={open}` 控制 React Query，不开启不请求，列表页面负荷不增加。

## 十三、v1.3 体验细化（UX Polish）

本期不引入新业务能力，聚焦于「核验—申报」全链路的可读性、复核效率与视觉一致性。

### 13.1 设计目标

1. **核验日志从「调试视图」升级为「复核视图」**：财务人员可凭日志独立完成事后复核。
2. **每个面板的语义更克制**：去除技术注解、Mock 痕迹，使界面用语贴近税务/财务专业语境。
3. **数字呈现彻底标准化**：解决金额换行、列宽参差、对齐错位等观感问题。
4. **保留 Tab 链条逻辑**：达人档案 → 双源验证 → 收入登记 → 税额试算 → 批量同步 → 申报导出 → 核验日志，对应财务月度作业的实际动作顺序,本期仅在该链条内做局部增强。

### 13.2 核验日志（重点改造）

#### 13.2.1 列表上方新增 6 项筛选

| 筛选项 | 字段 | 说明 |
| ---- | ---- | ---- |
| 关键字 | `talent_name` / `trace_id` | 模糊匹配,便于按达人或流水号定位 |
| 所属期 | `period` (YYYY-MM) | 月份控件 |
| 所得类型 | `income_type` | 复用 `INCOME_TYPES` 字典 |
| 签约关系 | `relation` | 复用 `RELATIONS` 字典 |
| 通过状态 | `passed` | 通过 / 不通过 / 全部 |
| 来源 | `source` | 单条核验 / 批量同步(CSV) / 批量同步(小红书) |

所有筛选基于内存数据进行 `useMemo` 过滤,不调用后端,避免在事后复核场景中频繁查询。

#### 13.2.2 列表展示业务字段

11 列业务视图:时间 / 达人 / 所属期 / 所得类型 / 签约关系 / 核验方式 / 来源 / 结果 / 收入金额 / 应纳税额 / 说明 + 流水号。

顶部新增统计行:总条数、通过/未通过分项、合计收入、合计税额。

去掉「最近 200 条」字样,后端默认返回上限提升至 1000 条,前端按需筛选。

#### 13.2.3 数据库字段扩展

`kyc_logs` 增加 7 列:

```
talent_name TEXT,
source TEXT,            -- single | batch | xhs-sync
period TEXT,
income_type TEXT,
relation TEXT,
amount REAL,
tax_amount REAL
```

这些字段在写入时由 `runBatch()` 与 `/api/verify` 路由统一冗余写入,避免事后复核时再做联表查询。新字段允许为空,旧记录平滑兼容。

### 13.3 各面板文案与语义微调

| 面板 | 调整 |
| ---- | ---- |
| 双源验证 | 去掉「Mock:基础格式校验 + 简易校验位算法」一句,改为「输入达人三/四要素信息可发起核验」 |

### 13.4 全系统数字样式统一

#### 13.4.1 货币格式化

`lib/format.ts` 重写 `fmtMoney`:

- ¥ 与数字之间使用不换行空格 `\u00a0` 拼接,**根本上避免符号与数字被拆到两行**。
- 新增 `fmtMoneyCompact` (大额时显示 万 / 亿)、`fmtCount` (千分位整数)、`fmtPercent` (百分比单位统一)。

#### 13.4.2 表格单元格规范

所有数字列遵循以下 Tailwind 组合:

```
text-right tabular-nums whitespace-nowrap
```

- `tabular-nums`: 等宽数字,跨行对齐整齐。
- `whitespace-nowrap`: 阻止单元格内换行。
- 强调列(税额、合计)追加 `font-semibold text-foreground`;辅助列使用 `text-muted-foreground`。
- 表格容器统一包 `<div className="overflow-x-auto">` 或 `CardContent` 加 `overflow-x-auto`,小屏自动横向滚动。

该标准已应用于:核验日志、收入登记、批量同步(2 张结果表)、申报导出、税额试算、达人详情抽屉。

### 13.5 架构与解耦原则延续

- 后端仅扩字段,不改业务计算流程;`runBatch()` 与 `verify` 路由仅做「冗余写日志」的轻动作。
- 前端筛选纯本地计算,不污染后端 API 形态。
- `lib/format.ts` 是 UI 层独立模块,任何面板替换均不影响业务模块。
